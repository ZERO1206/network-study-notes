# 全体系总复习 —— 五层模型串讲所有协议

---

## 一、五层骨架

```
┌─────────────────────────────────────────────┐
│ 应用层    HTTP  DNS  DHCP  SMTP/POP3/IMAP  FTP │  "我要什么"
├─────────────────────────────────────────────┤
│ 传输层              TCP          UDP          │  "怎么送过去"
├─────────────────────────────────────────────┤
│ 网络层           IP    ARP    ICMP            │  "走哪条路"
├─────────────────────────────────────────────┤
│ 链路层            MAC  交换机                 │  "一步怎么走"
├─────────────────────────────────────────────┤
│ 物理层          网线/WiFi/光纤                │  "信号怎么发"
└─────────────────────────────────────────────┘
```

---

## 二、应用层 —— 最贴近你的一层

| 协议 | 端口 | 传输 | 一句话 |
|------|------|------|--------|
| HTTP | 80 | TCP | 明文请求网页 |
| HTTPS | 443 | TCP | HTTP + TLS 加密 |
| DNS | 53 | UDP | 域名 → IP |
| DHCP | 67/68 | UDP | 自动分配 IP + 掩码 + 网关 + DNS |
| SMTP | 25 | TCP | 发邮件 |
| POP3 | 110 | TCP | 收邮件到本地 |
| IMAP | 143 | TCP | 多设备同步邮件 |
| FTP | 21/20 | TCP | 文件传输（双通道） |

### 核心概念

- **HTTP 无状态** → Cookie + Session 记住你是谁
- **DNS** 是上网第一步，没有它你连不上任何网站
- **DHCP** 给你四件套（IP、掩码、网关、DNS），缺一不可
- **邮件三协议** 分工明确：SMTP 发，POP3/IMAP 收

## 三、传输层 —— 数据分块 + 保证送达（或不保证）

| | TCP | UDP |
|------|-----|------|
| 连接 | 三次握手 | 无 |
| 可靠 | ACK + 重传 | 丢了不管 |
| 有序 | 序号重排 | 可能乱序 |
| 速度 | 慢（但有保障） | 快（但没保障） |
| 头部 | 20~60B | 8B |
| 场景 | 网页/文件/邮件 | 游戏/直播/DNS |

### TCP 可靠性四件套

```
ACK + 重传      →  丢了就重发
序号           →  乱了就重排
流量控制        →  接收方别撑死
拥塞控制        →  网络别堵死
```

### 拥塞控制三阶段

```
慢启动(指数×2) → 达 ssthresh → 拥塞避免(线性+1) → 丢包 → 砍半/回退
```

- 超时丢包 = 严重 → cwnd 回到 1，重新慢启动
- 3 个重复 ACK = 不严重 → cwnd 砍半，快速恢复（Reno）

### 核心概念

- 端口号 = 区分同一台机器上的不同程序
- IP + 端口 = 全球唯一标识某个进程
- 三次握手建连，四次挥手断连
- TCP 保送到，UDP 保速度

## 四、网络层 —— 寻址 + 路由

**主角：IP 协议。** 把数据从 A 逐跳转发到 B。不可靠（只管尽力转发），可靠是 TCP 的事。

### 配角

| 协议 | 问题 | 一句话 |
|------|------|--------|
| ARP | 知道 IP，不知道 MAC | IP → MAC（局域网广播） |
| ICMP | 怎么报告网络状态 | Ping/Traceroute 靠它 |
| NAT | 私有 IP 怎么上公网 | 内网 IP+端口 ↔ 公网 IP+端口 |

### IP 地址核心知识

```
传统分类（考试考）：
  A 类 1~126    /8      0.......
  B 类 128~191  /16     10......
  C 类 192~223  /24     110.....

现实用 CIDR：192.168.1.0/24 ← 前缀任意长

子网划分三步：
  ① 借位：分 N 个子网 = 2^m → 借 m 位
  ② 步长：256 − 掩码末字节（或 2^剩余主机位）
  ③ 可用：每子网 − 2（网络地址 + 广播地址）
```

**IPv6：** 128 位，冒号十六进制，`::` 压缩全零，头部固定 40B

## 五、链路层 + 物理层

- MAC 地址 = 网卡身份证，出厂烧录，交换机据此转发
- ARP 在 IP 和 MAC 之间搭桥
- 物理层负责电信号/光信号/无线电波

## 六、完整数据旅程

```
你在浏览器输入 baidu.com 按下回车：

① DNS(UDP,端口53)
   域名 baidu.com → IP 110.242.68.66

② 子网掩码判断
   不在同一网段 → 走默认网关

③ ARP 广播
   "网关的 MAC 是多少？" → 拿到网关 MAC

④ TCP 三次握手
   SYN → SYN+ACK → ACK → 连接建立

⑤ HTTP 发送请求
   GET / HTTP/1.1
   Host: www.baidu.com

⑥ 服务器响应
   HTTP/1.1 200 OK
   Content-Type: text/html
   <html>...</html>

⑦ TCP 四次挥手
   FIN → ACK → FIN → ACK → 连接关闭
```

### 每层做了什么

```
应用层：构造 HTTP 请求，先找 DNS 翻译域名
传输层：TCP 分段 + 端口号，三次握手建连
网络层：IP 封装 + 逐跳转发，ARP 找 MAC
链路层：帧封装 + MAC 地址，交换机转发
物理层：0101... → 电信号/光信号 → 物理传输
```

```
                   只拆到网络层，看 IP 转发
```

## 自测题

<details>
<summary><b>1. 五层模型每层一句话说清楚职责</b></summary>

> **应用层**：为应用程序提供网络服务（HTTP/DNS/DHCP）
>
> **传输层**：数据分段、端口寻址、可靠/不可靠传输（TCP/UDP）
>
> **网络层**：IP 寻址 + 逐跳路由转发（IP/ARP/ICMP）
>
> **链路层**：相邻设备间按 MAC 地址传输帧
>
> **物理层**：把 0101 转成电信号/光信号/无线电波

</details>

<details>
<summary><b>2. 输入网址到网页返回，每一步发生了什么？</b></summary>

> ```
> ① DNS    → 域名 → IP（UDP 端口53）
> ② 子网掩码 → 判断是否同网段，不同则走网关
> ③ ARP    → 找到网关 MAC 地址
> ④ TCP    → 三次握手（SYN → SYN+ACK → ACK）
> ⑤ HTTP   → 发送 GET / HTTP/1.1，Host: xxx
> ⑥ 服务器  → 返回 HTTP/1.1 200 OK + HTML
> ⑦ TCP    → 四次挥手（FIN → ACK → FIN → ACK）
> ```

</details>

<details>
<summary><b>3. TCP 怎么保证可靠？（四个机制）</b></summary>

> 1. **ACK + 重传**：收到确认，超时未收到 ACK 就重发
> 2. **序号**：每个字节编号，接收方按序号重排，不乱序
> 3. **流量控制**：接收方告知窗口大小，"我只能收这么多"
> 4. **拥塞控制**：发送方感知网络拥堵（丢包），自动减速

</details>

<details>
<summary><b>4. TCP 为什么三次握手、四次挥手？</b></summary>

> **三次握手（建连）**：防止旧的 SYN 请求让服务器白白建立连接。第三次 ACK 让客户端确认"我确实要连"。
>
> ```
> 两次的话：客户端发 SYN → 服务器回 SYN+ACK → 服务器就认为连上了
> 但旧 SYN 延迟到达时，服务器会建立废弃连接
> ```
>
> **四次挥手（断连）**：TCP 是全双工，两个方向各自关闭。客户端说完了 ≠ 服务器说完了。
>
> ```
> 客户端 FIN → 服务器 ACK → 服务器 FIN → 客户端 ACK
> 第二、三步不能合并：服务器可能还有数据没传完
> ```

</details>

<details>
<summary><b>5. NAT 的转换表长什么样？为什么要端口映射？</b></summary>

> **转换表：**
>
> ```
> 内网地址:端口           →    公网地址:端口
> 192.168.1.10:54321    →    1.2.3.4:40001
> 192.168.1.11:54321    →    1.2.3.4:40002
> ```
>
> **为什么要端口映射：** 全家多台设备共享同一个公网 IP，回包来了得知道还给内网哪台设备。靠端口号区分——即使两台用了相同源端口，NAT 映射成不同公网端口，查表分拣。
>
> **副作用：** 外面无法主动连进来，NAT 表里没记录就丢弃。

</details>

<details>
<summary><b>6. 子网划分：192.168.1.0/24 分 4 个子网，写出每个的范围</b></summary>

> ```
> 4 = 2² → 借 2 位 → 新前缀 /26 → 掩码 255.255.255.192
> 步长 = 256 − 192 = 64
>
> 子网1: 192.168.1.0/26   (.1 ~ .62)     网络 .0,   广播 .63
> 子网2: 192.168.1.64/26  (.65 ~ .126)   网络 .64,  广播 .127
> 子网3: 192.168.1.128/26 (.129 ~ .190)  网络 .128, 广播 .191
> 子网4: 192.168.1.192/26 (.193 ~ .254)  网络 .192, 广播 .255
>
> 每子网 64 地址，可用 62（减网络地址 + 广播地址）
> ```

</details>

<details>
<summary><b>7. HTTPS 混合加密为什么不用纯非对称？</b></summary>

> 非对称加密安全但**太慢**（比对称慢几百倍），不适合加密大量网页数据。
>
> 实际方案：**非对称传钥匙，对称传内容。**
>
> ```
> ① 服务器给公钥（非对称）
> ② 客户端生成临时对称密钥
> ③ 用公钥加密对称密钥发过去（非对称，就这一小坨）
> ④ 之后全部用对称密钥通信（快）
> ```
>
> 数字证书（CA 签名）保证公钥是真的，不是中间人伪造的。

</details>

<details>
<summary><b>8. DNS 走 UDP，HTTP 走 TCP —— 为什么？</b></summary>

> **DNS 用 UDP：**
> - 一次查询很小（几百字节），不值得三次握手建连接
> - 要求极快，域名解析是上网第一步，慢了拖慢一切
> - 丢了无所谓，有超时重试机制，换一个 DNS 服务器再问
>
> **HTTP 用 TCP：**
> - 网页内容不能丢（丢了整个页面就可能残缺或加载失败）
> - 需要可靠、完整、有序的传输
> - 文件一般较大，TCP 流量控制和拥塞控制是必要的

</details>

<details>
<summary><b>9. 流量控制 vs 拥塞控制，区别在哪？</b></summary>

> | | 流量控制 | 拥塞控制 |
> |------|---------|----------|
> | 问题根源 | 接收方处理不过来 | **网络本身堵了** |
> | 谁告知 | 接收方（窗口字段） | 发送方自己**猜**（丢包≈拥塞） |
> | 手段 | 接收方说"我只能收 xxx" | 发送方探测/减速 |
>
> 两者共同决定实际窗口：`min(接收方窗口, 拥塞窗口)`

</details>

<details>
<summary><b>10. 私有 IP 三段分别是什么？为什么需要私有 IP？</b></summary>

> ```
> 10.0.0.0/8      → 10.x.x.x        （公司/学校）
> 172.16.0.0/12   → 172.16~31.x.x   （较少见）
> 192.168.0.0/16  → 192.168.x.x     （家庭路由器）
> ```
>
> **为什么需要：** IPv4 地址不够，全用公网 IP 早就没了。私有 IP 在局域网内可以**重复使用**（全球无数家庭都用 192.168.1.x），再通过 NAT 共享一个公网 IP 上网。
</details>
