# HTTP 深入

---

## 一、请求方法

| 方法 | 含义 | 类比 |
|------|------|------|
| GET | 获取资源（查） | "服务员，看菜单" |
| POST | 提交数据/新建（增） | "我要下单" |
| PUT | 整条更新（改） | "这份换掉" |
| DELETE | 删除（删） | "这份不要了" |

最常见就 GET 和 POST。GET 数据拼在 URL 里（`?name=张三&age=18`），POST 放在请求体里，相对安全。

---

## 二、Cookie 与 Session

### HTTP 是无状态的

每次请求都是独立的，服务器不记得你上一条干了什么。这很要命——登录后每点一个页面都得重新输密码。

### Cookie

服务器在响应里写 `Set-Cookie`，浏览器保存后**每次请求自动带上**：

```
服务器响应：
Set-Cookie: user_id=abc123

之后每次请求自动带上：
Cookie: user_id=abc123
```

就像水吧领的手环，之后每次刷手环就行，不用掏身份证。

### Session

Cookie 里不直接存敏感信息。服务器端存真正的用户数据叫 **Session**，Cookie 里只放一个 Session ID：

```
Cookie: session_id=x7k3m9   ← 只是一个编号
                    ↕
服务器 Session 表：
  x7k3m9 → { user_id: 123, name: "张三", login: true }
```

**浏览器只拿编号，实际数据在服务器手里。** 安全得多。

## 三、HTTP 缓存

### 强缓存

服务器告诉浏览器："这个资源 7 天内不要再来问我了"：

```
Cache-Control: max-age=604800   ← 7天内直接用本地缓存
```

### 协商缓存

缓存过期后，浏览器先问："没换吗？还要吗？"：

```
浏览器：If-Modified-Since: 上周日期
服务器："304 Not Modified" ← 没变，用缓存
        "200 OK"          ← 变了，给你新的
```

**304 = 没变，用缓存的就行。**

---

## 关键点回顾

- GET 获取，POST 提交 → 数据位置不同（URL vs 请求体）
- HTTP 无状态 → Cookie 存 Session ID → 服务器查 Session 确认身份
- 强缓存不发请求，协商缓存发请求但可能不传内容（304）
