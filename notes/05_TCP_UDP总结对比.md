# TCP 与 UDP 总结对比

---

## 核心差异对比

| | TCP | UDP |
|------|-----|------|
| 面向连接 | 是（三次握手） | 否（直接发） |
| 可靠性 | 可靠（ACK + 重传） | 不可靠 |
| 有序性 | 有序（序号重排） | 可能乱序 |
| 传输速度 | 较慢 | 快 |
| 头部大小 | 20~60 字节 | 8 字节 |
| 流量控制 | 有（滑动窗口） | 无 |
| 拥塞控制 | 有（慢启动/拥塞避免） | 无 |
| 适用场景 | 网页/文件/邮件 | 游戏/直播/语音/DNS |

---

## 各自典型应用

```
TCP 场景（数据不能丢）：
├─ HTTP/HTTPS → 网页浏览
├─ FTP        → 文件传输
├─ SMTP/POP3  → 邮件
└─ SSH/Telnet → 远程登录

UDP 场景（实时性优先）：
├─ DNS        → 快速域名查询
├─ DHCP       → 获取 IP 地址
├─ VoIP       → 语音视频通话
├─ 在线游戏   → 位置状态更新
├─ 直播/流媒体 → 实时音视频
└─ TFTP/SNMP  → 简单网络管理
```

---

## 怎么选？

```
数据一丁点都不能丢？
  是 → TCP
  否 → 延迟比丢包更致命？
          是 → UDP（游戏、通话、直播）
```

## 面试/考研常见考点

<details>
<summary><b>1. TCP 和 UDP 的区别是什么？（100% 必问）</b></summary>

> **回答要点：**
>
> TCP 面向连接、可靠、有序，通过三次握手建立连接，ACK + 序号保证不丢不乱，有流量控制和拥塞控制，头部 20~60 字节，适合网页/文件/邮件。
>
> UDP 无连接、不可靠、可能乱序，发了不管，头部仅 8 字节，没有流量和拥塞控制，但速度极快，适合游戏/直播/语音/DNS。
>
> **一句话：TCP 保送到，UDP 保速度。**

</details>

<details>
<summary><b>2. TCP 如何保证可靠传输？</b></summary>

> **回答要点：**
>
> - **ACK + 重传**：接收方确认收到，发送方超时未收到 ACK 就重发
> - **序号**：每个字节编号，接收方按序号重排，不乱序
> - **流量控制**：接收方告知自己能收多少（滑动窗口），防止发送方太快
> - **拥塞控制**：发送方感知网络拥堵（丢包），自动减速（慢启动/拥塞避免）
>
> 这四者共同保证了 TCP 的可靠性。

</details>

<details>
<summary><b>3. 为什么 DNS 用 UDP 而不是 TCP？</b></summary>

> **回答要点：**
>
> 1. DNS 查询的数据量很小（一次请求+响应通常几百字节），不值得为它建立 TCP 连接（三次握手开销大）
> 2. DNS 查询要求**快**，域名解析是上网第一步，慢了解析会拖慢整个网页打开速度
> 3. 丢了无所谓——DNS 有超时重试机制，一次查不到换一个 DNS 服务器再问就行
>
> **注：** DNS 在某些场景也用 TCP（如区域传输、响应超过 512 字节时）

</details>

<details>
<summary><b>4. 视频通话/在线游戏为什么用 UDP？</b></summary>

> **回答要点：**
>
> 1. **实时性 > 可靠性**：丢一帧画面/一个位置包，下一秒新的数据就来了，旧数据重传没有意义
> 2. TCP 重传会导致**延迟累积**——一个包丢了等重传，后面的包全堵着，画面卡顿比丢几帧难受得多
> 3. 语音/视频编码本身有一定的**丢包容忍度**，丢少量数据不影响听懂/看清

</details>

<details>
<summary><b>5. TCP 三次握手能不能用 UDP？为什么 UDP 不需要握手？</b></summary>

> **回答要点：**
>
> 不能。TCP 需要握手是因为它要建立**双方都确认的可靠连接**——交换初始序号、协商参数、确认对方在听。
>
> UDP 不需要握手正是因为它**不保证可靠**——你发你的，我不管你是不是在听，丢了也不重传。没有"连接"这个概念，自然不需要握手的开销。

</details>

---

## 关键点回顾

- TCP 头部 20~60 字节（处理多），UDP 头部 8 字节（轻量）
- DNS 用 UDP，HTTP 用 TCP
- UDP 适合：丢部分数据无所谓，延迟比丢包更致命的场景
