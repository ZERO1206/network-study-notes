# HTTPS 与 TLS 加密

---

## 问题

HTTP 全是明文。任何中间节点都能看到你发了什么、服务器回了什么。公共 WiFi 上输密码 = 裸奔。

**HTTPS = HTTP + TLS/SSL（加密层）**

---

## 三种加密方式

### 1. 对称加密 —— 快，但密钥怎么给对方？

同一把钥匙既加密又解密。效率极高，但**密钥本身怎么安全传递**是无解的——公开信道直接传密钥，中间人拿到就全破了。

### 2. 非对称加密 —— 安全，但太慢

**公钥加密，私钥解密。** 公钥可以公开发给任何人。

```
任何人用公钥加密 → 只有私钥持有者能解开
```

解决了密钥传输问题，但比对称加密慢几百倍，不适合加密大量数据。

### 3. 实际方案：混合加密

```
① 服务器先把公钥给客户端（非对称）
② 客户端生成"临时对称密钥"
③ 用公钥加密临时密钥，发给服务器（非对称，就加密这一小坨）
④ 服务器用私钥解密，拿到临时密钥
⑤ 之后全程用临时密钥对称加密传输（快）
```

**非对称传钥匙，对称传内容。各取所长。**

## TLS 握手简版

```
客户端                                    服务器
   │                                        │
   ├── ①"你好，我支持这些加密方式"───────→│
   │                                        │
   │←── ②"选AES，这是我的证书(含公钥)"────┤
   │                                        │
   ├── ③验证证书(CA签名)                  │
   │      生成对称密钥                      │
   │      用公钥加密对称密钥 ──────────────→│
   │                                        │
   │←── ④"OK，切换加密"──────────────────┤
   │                                        │
   │←── 全程对称加密通信 ────────────────→│
```

## 数字证书 —— 公钥是真的吗？

攻击者可以在中间伪造公钥，偷看内容。

**数字证书 = 公钥的"身份证"**，由 CA（证书颁发机构）签发：

```
你的浏览器内置了可信 CA 列表
  ↓
服务器把信息和公钥发给 CA → CA 用自己的私钥签名 → 生成证书
  ↓
浏览器用 CA 的公钥验签名 → 通过 = 公钥可信 → 地址栏小锁
```

**证书解决的核心问题：确认"这把公钥确实是你的，不是中间人伪造的"。**

---

## 关键点回顾

- HTTPS = HTTP + TLS/SSL
- 对称加密快但密钥不好传，非对称安全但慢 → **混合用**
- 非对称传对称密钥，对称加密传内容
- 数字证书 = CA 背书的公钥身份证，防中间人伪造
- 浏览器小锁 = 证书验证通过
