# TCP 拥塞控制

---

## 流量控制 vs 拥塞控制

| | 流量控制 | 拥塞控制 |
|------|---------|----------|
| 问题根源 | 接收方处理不过来 | **网络本身堵了** |
| 谁告知 | 接收方（窗口字段） | 发送方自己**猜** |
| 手段 | 接收方说"我只能收 xxx" | 发送方探测/调整速度 |

流量控制是端到端的，拥塞控制是发送方对整条网络路径的感知。

---

## 怎么感知拥塞？

TCP 没有交警告诉它堵不堵。靠一个简单指标：**丢包 ≈ 网络堵了。**

中间路由器队列满了 → 丢包。链路出错很少，所以丢包基本等于拥塞。

---

## 拥塞窗口（cwnd）

发送方维护一个**拥塞窗口（cwnd）**——当前允许飞在路上的最大数据量：

```
实际窗口 = min(接收方通告窗口, 拥塞窗口)
```

谁更严听谁的。

---

## 慢启动 —— 别一上来就把路堵死

刚建立连接时不知道网络能承载多少，从很小开始，**指数增长**：

```
cwnd = 1 MSS（约 1460 字节）

发 1 个 → 收到 ACK → cwnd × 2 = 2
发 2 个 → 收到 ACK → cwnd × 2 = 4
发 4 个 → 收到 ACK → cwnd × 2 = 8
发 8 个 → 收到 ACK → cwnd × 2 = 16
...
```

叫"慢"启动但实际是**指数爆炸**——很快就上去了。

---

## 拥塞避免 —— 快到了就别猛冲

cwnd 涨到 **ssthresh（慢启动门限）** 后，改成**线性增长**：

```
cwnd < ssthresh  →  慢启动（指数 ×2）
cwnd >= ssthresh →  拥塞避免（每RTT +1 MSS）
```

```
cwnd
 ↑
 │    ╱                     ← 慢启动（指数）
 │  ╱
 │ ╱
 │╱
 │  ─────────────────────── ssthresh
 │                           ╲_____  ← 拥塞避免（线性）
 └──────────────────────────────→ 时间
```

**指数增长太快，极容易直接冲过量造成大量丢包。线性慢慢试探边界。**

---

## 丢包后的处理

### TCP Tahoe（老版本）
```
丢包 → ssthresh = cwnd/2, cwnd = 1 → 重新慢启动
```
简单粗暴，丢一次回到解放前。

### TCP Reno（更常用）
```
超时丢包        → ssthresh = cwnd/2, cwnd = 1 → 慢启动
3个重复ACK丢包  → ssthresh = cwnd/2, cwnd = cwnd/2 → 拥塞避免（快速恢复）
```
超时 = 严重拥堵，退回起点。
重复 ACK = 丢了一个但后面还在到 → 砍一半，从一半继续。

---

## 整体流程

```
连接建立 → 慢启动(×2) → 达ssthresh → 拥塞避免(+1)
                ↑                        ↓
                └──── 丢包超时 ←─────────┘
```

## 类比：往水管灌水

```
不知道水管多粗 → 小水流试 → 慢慢加大
水溢出(丢包)  → 关小一半 → 再慢慢加大
循环：探测 → 拥塞 → 回退 → 探测 → ...
```

---

## 关键点回顾

- 流量控制：接收方不行；拥塞控制：**网络不行**
- 慢启动 = **指数增长**，拥塞避免 = **线性增长**
- 丢包 = 拥塞信号 → cwnd 砍半（或回 1）
- Reno 比 Tahoe 更聪明：重复 ACK 不完全回退
