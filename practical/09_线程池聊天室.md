# 第九课：线程池聊天室 —— epoll + 多线程并行

> 把第七课的 epoll 聊天室升级为 epoll + 线程池，主线程只管收消息，工作线程并行广播。

---

## 一、为什么有了 epoll 还需要线程池

epoll 解决了"I/O 等待不阻塞"，但没解决"CPU 处理不排队"。

```
epoll 单线程版：
  epoll_wait → recv A的消息 → 广播给 1000 人（999次send）
                                         ↑
                             这段时间 B 的消息必须排队等 A 处理完

epoll + 线程池：
  主线程：epoll_wait → recv → 扔任务进队列 → 立刻回到 epoll_wait
  工作线程1：取任务 → 广播 A 的消息
  工作线程2：取任务 → 广播 B 的消息    ← 并行处理
  工作线程3：取任务 → 广播 C 的消息
```

---

## 二、线程池结构

```
                               ┌─ 工作线程1
主线程 ──push任务──→ 任务队列 ──┼─ 工作线程2
  ↑                   ↑        ├─ 工作线程3
  │                   │        └─ 工作线程4
  │            mutex + cond_var
  │                   │
  └── signal 唤醒 ────┘
```

| 组件 | 类型 | 作用 |
|------|------|------|
| `task_queue` | `std::queue<Task>` | 存放待执行的任务 |
| `queue_lock` | `pthread_mutex_t` | 保护队列，防止多线程同时 push/pop |
| `queue_cond` | `pthread_cond_t` | 条件变量——队列空时工作线程睡觉，有任务时叫醒 |
| `running` | `bool` | 线程池开关，程序退出时设为 false |

---

## 三、核心机制详解

### 3.1 工作线程 —— 取任务 + 执行

```cpp
void* worker_thread(void* arg) {
    (void)arg;
    while (true) {
        Task task;

        pthread_mutex_lock(&queue_lock);
        while (running && task_queue.empty()) {
            pthread_cond_wait(&queue_cond, &queue_lock);  // 睡觉等任务
        }

        if (!running && task_queue.empty()) {
            pthread_mutex_unlock(&queue_lock);
            break;   // 程序要退出，队列也清空了
        }

        task = task_queue.front();
        task_queue.pop();
        pthread_mutex_unlock(&queue_lock);

        task();   // 在锁外面执行！不阻塞其他线程
    }
    return nullptr;
}
```

**为什么用 `while` 而不是 `if` 检查队列空：** `cond_wait` 可能被虚假唤醒（内核信号干扰）。醒来后用 `while` 再检查一次，队列还是空就继续睡。

**为什么 `task()` 在锁外面执行：**

```
✅ 正确：                       ❌ 错误：
lock → pop 任务 → unlock      lock → pop 任务
  ↓                              ↓
执行 task（不加锁）              执行 task（持锁中）
  ↓                              ↓
下次取任务时再加锁               其他线程干等 lock
                                 ↑ 几百毫秒！
```

持锁时执行任务 → 任务里的 send/recv 可能耗时 → 所有线程被挡在锁外面 → 线程池退化回串行。

### 3.2 条件变量 —— 工作线程怎么知道有任务

```
工作线程：pthread_cond_wait(&cond, &lock)
         → 原子操作：释放锁 + 进入睡眠
         → 被 signal 唤醒 → 自动重新获取锁 → 返回

主线程：pthread_cond_signal(&cond)
       → 唤醒一个正在 wait 的线程
```

**`cond_wait` 醒来时锁已经在手上了。** POSIX 规范保证：释放锁 → 睡眠 → 被唤醒 → 重新获取锁 → 返回。所以 cond_wait 返回后可以直接安全访问共享变量。

### 3.3 主线程提交任务

```cpp
// recv 拿到消息后
pthread_mutex_lock(&queue_lock);
task_queue.push([&clients, client_fd, msg]() {
    // lambda 捕获：clients引用、client_fd值、msg值
    pthread_mutex_lock(&queue_lock);
    for (int other : clients) {
        if (other != client_fd) {
            send(other, msg.c_str(), msg.size(), 0);
        }
    }
    pthread_mutex_unlock(&queue_lock);
});
pthread_cond_signal(&queue_cond);   // 叫醒一个工作线程
pthread_mutex_unlock(&queue_lock);
```

**Lambda 捕获说明：**

| 捕获 | 方式 | 为什么 |
|------|------|--------|
| `clients` | 引用 `&` | 广播需要遍历所有在线用户 |
| `client_fd` | 值 | 发送者 fd（不变） |
| `msg` | 值 | 消息内容已拷出 `buf`，不依赖原缓冲区 |

### 3.4 每个任务独立缓冲区

```cpp
char* buf = new char[1024];     // 每次 recv 分配新缓冲区
int n = recv(client_fd, buf, 1023, 0);
// ...
std::string msg(buf, n);        // 拷出内容
delete[] buf;                   // 立即释放
```

不能共用一块 `char buf[4096]`——下一个客户端发来消息时，上一个任务的 `buf` 就被覆盖了。`new + delete` 保证每个任务拿到自己的数据。

## 四、完整对比：三代聊天室

| | 第一代 select | 第二代 epoll | 第三代 epoll + 线程池 |
|------|-------------|------------|---------------------|
| I/O 模型 | select | epoll_wait | epoll_wait |
| 并发模型 | 单线程串行 | 单线程串行 | **多线程并行** |
| 广播谁做 | 主线程自己做 | 主线程自己做 | **线程池并行广播** |
| 瓶颈 | O(n) 扫描 + 1024 上限 | 广播时阻塞其他客户端 | 广播并行，互不阻塞 |
| fd 上限 | 1024 | 无限制 | 无限制 |
| 共享变量保护 | 不需要 | 不需要 | **需要 mutex 保护** |

## 五、完整代码

```cpp
#include <iostream>
#include <cstring>
#include <vector>
#include <sys/socket.h>
#include <netinet/in.h>
#include <unistd.h>
#include <sys/epoll.h>
#include <pthread.h>
#include <queue>
#include <functional>

using Task = std::function<void()>;
std::queue<Task> task_queue;
pthread_mutex_t queue_lock = PTHREAD_MUTEX_INITIALIZER;
pthread_cond_t  queue_cond = PTHREAD_COND_INITIALIZER;
bool running = true;

void* worker_thread(void* arg) {
    (void)arg;
    while (true) {
        Task task;

        pthread_mutex_lock(&queue_lock);
        while (running && task_queue.empty()) {
            pthread_cond_wait(&queue_cond, &queue_lock);
        }

        if (!running && task_queue.empty()) {
            pthread_mutex_unlock(&queue_lock);
            break;
        }

        task = task_queue.front();
        task_queue.pop();
        pthread_mutex_unlock(&queue_lock);

        task();
    }
    return nullptr;
}

int main() {
    int server_fd = socket(AF_INET, SOCK_STREAM, 0);

    int opt = 1;
    setsockopt(server_fd, SOL_SOCKET, SO_REUSEADDR, &opt, sizeof(opt));

    sockaddr_in addr{};
    addr.sin_family = AF_INET;
    addr.sin_addr.s_addr = INADDR_ANY;
    addr.sin_port = htons(9999);
    bind(server_fd, (sockaddr*)&addr, sizeof(addr));
    listen(server_fd, 10);

    std::cout << "[线程池聊天室] 监听 0.0.0.0:9999\n";

    std::vector<int> clients;

    int epoll_fd = epoll_create1(0);
    struct epoll_event ev;
    ev.events = EPOLLIN;
    ev.data.fd = server_fd;
    epoll_ctl(epoll_fd, EPOLL_CTL_ADD, server_fd, &ev);
    struct epoll_event events[1024];

    const int THREAD_COUNT = 4;
    for (int i = 0; i < THREAD_COUNT; ++i) {
        pthread_t tid;
        pthread_create(&tid, nullptr, worker_thread, nullptr);
        pthread_detach(tid);
    }

    while (true) {
        int n = epoll_wait(epoll_fd, events, 1024, -1);

        for (int i = 0; i < n; ++i) {
            int fd = events[i].data.fd;

            if (fd == server_fd) {
                int client_fd = accept(server_fd, nullptr, nullptr);
                ev.events = EPOLLIN;
                ev.data.fd = client_fd;
                epoll_ctl(epoll_fd, EPOLL_CTL_ADD, client_fd, &ev);
                clients.push_back(client_fd);
                std::cout << "[聊天室] 新用户 fd=" << client_fd << "\n";
            }
            else {
                int client_fd = fd;
                char* buf = new char[1024];
                int n = recv(client_fd, buf, 1023, 0);

                if (n <= 0) {
                    delete[] buf;
                    epoll_ctl(epoll_fd, EPOLL_CTL_DEL, client_fd, nullptr);
                    close(client_fd);
                    pthread_mutex_lock(&queue_lock);
                    for (auto it = clients.begin(); it != clients.end(); ++it) {
                        if (*it == client_fd) { clients.erase(it); break; }
                    }
                    pthread_mutex_unlock(&queue_lock);
                    std::cout << "[聊天室] 离开 fd=" << client_fd << "\n";
                }
                else {
                    buf[n] = '\0';
                    std::string msg(buf, n);

                    pthread_mutex_lock(&queue_lock);
                    task_queue.push([&clients, client_fd, msg]() {
                        pthread_mutex_lock(&queue_lock);
                        for (int other : clients) {
                            if (other != client_fd) {
                                send(other, msg.c_str(), msg.size(), 0);
                            }
                        }
                        pthread_mutex_unlock(&queue_lock);
                    });
                    pthread_cond_signal(&queue_cond);
                    pthread_mutex_unlock(&queue_lock);

                    delete[] buf;
                }
            }
        }
    }

    close(server_fd);
    return 0;
}
```

编译：`g++ -std=c++17 -o threadpool_server threadpool_server.cpp -lpthread`

## 六、当前缺陷与改进方向

### 1. 广播持锁执行 I/O

lambda 里持锁时执行 `send`，少数慢客户端可能影响其他线程取锁。

**改进方向：** 每个客户端独立发送缓冲区 + 非阻塞 send，epoll 负责可写事件。

### 2. epoll_wait 每次传 1024 事件上限

`events[1024]` 定死容量，超过 1024 个同时有动静就漏事件。

**改进方向：** events 数组动态扩容，或增大容量。

### 3. 断连时主线程直接操作

断连的 epoll_ctl + close + erase 在主线程里串行做，如果 erase 时 vector 很大可能耗时。

**改进方向：** 断连也交给线程池。

### 4. 无客户端超时

keep-alive 连接没有超时踢人，僵尸 fd 一直占着。

**改进方向：** 用 `epoll_wait` 超时参数 + 每个连接上次活动时间。

## 七、自测题

1. 有了 epoll 为什么还需要线程池？epoll 解决了什么没解决什么？
2. `pthread_cond_wait` 醒来时锁在手上吗？为什么？
3. 为什么 `task()` 要在锁外面执行？
4. 为什么检查队列空用 `while` 而不是 `if`？
5. `detach` 和 `join` 有什么区别？线程池里为什么用 `detach`？
6. 每个任务为什么要 `new char[1024]` 而不是共用一块缓冲区？

<details>
<summary>点击查看答案</summary>

1. epoll 解决了"I/O 等待不阻塞"——一个线程能同时监控多个 fd。但消息处理（广播）还是串行的——一个人的广播完才能处理下一个人的。线程池让多个广播并行执行，互不阻塞。

2. **一定在手上。** POSIX 规范：`cond_wait` = 原子释放锁 + 睡眠；被 signal 唤醒后自动重新获取锁；返回时锁已持有。所以返回后能安全访问共享变量。

3. 持锁时执行 `task()` → 任务里可能有耗时的 `send`/`recv` → 其他工作线程全被挡在锁外面 → 线程池退化回串行。锁只保护队列操作（push/pop），任务的执行是独立的。

4. `cond_wait` 可能被**虚假唤醒**（内核信号干扰、bug）。`if` 被虚假唤醒后直接往下执行，队列可能是空的。`while` 醒来后再检查一次，空就继续睡。

5. `join` 阻塞等待线程结束 + 回收资源。`detach` 立即返回，线程结束后内核自动回收。线程池用 `detach`——主线程要快速回到 `epoll_wait`，不能等。

6. 如果共用一块 `char buf[4096]`，上一个任务还没处理完，下一个客户端发来消息就把 `buf` 覆盖了。每个任务分配独立缓冲区，各自持有自己的数据副本。

</details>
